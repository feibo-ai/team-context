4 · 关键判断

判断:无感分发实现前过三轮独立对抗安全评审(评审通过前不得 code)
背景 ⑨⑩ 是代码执行面(daemon 静默换二进制 / 写可执行 skill = RCE)。先 vibe code 实现,还是先出专项安全设计评审?
选项 直接 vibe code 实现(快,但安全自审=同 session 既写又审,违 SOP) / 先出专项安全 mini-ADR + 过第二 session 独立对抗安全评审才 code
选择 ②。mini-ADR v1→v2→v3→v4 各经独立 4-lens 对抗评审 panel + synthesizer,逐条核对真实代码。
事后看 对,且是本项目最关键的判断。v1 揪出循环信任根(server 失陷可换公钥)、v2 揪长期裸私钥单点(外泄=离线无限伪造)、v3 揪散文缓解未落可验证 MUST。任一未挡,都会 ship 一个『已验证』章盖在未评审/未验证工件上——这比 SHA-256-only 更危险(给人安全错觉)。三轮严重度递降(5→3→0 新架构 Blocker)证明评审真在收敛而非空转。
古法不可能 是。每轮并行跑 4 个对抗 lens(供应链/密码学/字节确定性/运行时写路径)+ 主审 synthesizer,逐条对照真实代码核验声称,单人/传统流程做不到这个评审密度与速度。
判断:签名选型 keyless OIDC attestation,弃 minisign 长期私钥
背景 签名机制选型:长期私钥(minisign 单 keyset)vs GitHub keyless OIDC build-provenance attestation。
选项 minisign:离线简单、单 keyset,但需长期保管私钥(单点) / keyless OIDC attestation:无长期私钥,签名身份=CI workflow 的 OIDC subject
选择 ②keyless。长期私钥外泄=离线无限伪造的单点失陷;keyless 把签名绑死在 2-review-merged 态触发的 workflow OIDC 身份,无钥可窃。
事后看 对。E2E 实证:SAN 三元组(repo+workflow+ref)离线可验、daemon 内置 sigstore trusted root 验签 fail-closed、拒 upstream 同名工件。
古法不可能 否——机制选型本身是工程判断;但由多轮对抗评审驱动收敛到此选型(v1 minisign→v4 keyless)是 AI-native 评审密度的产物。
判断:team-context 转公开,解 attestation 的 org-plan 门控
背景 真实 E2E 测试撞到:私有仓的 GitHub artifact attestation 被 feibo-ai org 当前 plan 门控(报错明示 upgrade 或转公开)。
选项 升级 feibo-ai org 付费 plan(私有仓 attestation) / cosign keyless 另起 toolchain(私有可用但换栈) / skill 源仓转公开
选择 ③转公开(先扫 119 commit 全历史确认 0 token/key/secret 才转)。一举解两约束:signing 免费可用 + 公有仓 daemon 可直接轮询 release(省掉 server 不可信传输层)。
事后看 对。全链 live E2E(FetchLatestSkillBundle→Verify→Extract 24 文件 byte-for-byte)跑通证明。诚实前置:转公开前扫历史防凭据泄露,是不可逆动作的必要尽调。
古法不可能 是。是 live E2E 真实发版测试『经验性』撞出 plan 门控这个设计前提,而非纸上推演——快速真实基建迭代暴露了纸面设计漏掉的约束。
判断:daemon 写 ~/.claude/skills = 净新 RCE 面,单独威胁建模写路径安全
背景 daemon 原本只读上报、不写盘;⑩ 让它静默写可执行 skill 到用户 dotfile = 净新攻击面,比手动 pull 大。
选项 验签过就直接写盘(信任验签即可) / 把净新写路径当独立 RCE 面单独威胁建模 + 写路径安全 + 真攻击测试
选择 ②。tar typeflag 白名单(拒 symlink/hardlink entry)+ 路径穿越拒 + 父链逐级 EvalSymlinks(拒父目录软链逃逸)+ O_EXCL|O_NOFOLLOW 写原语二分 + 软链目标确定性拒写(非 follow 非静默跳过)+ 互斥守卫(dev 软链机拒启)。
事后看 对。W2/W3/W5 用真攻击布局测试(预置软链/父链软链/../穿越),不仅断言 error,更断言『victim 文件未被改写、外部目录未被写入』——证明攻击未得逞而非仅返回错误。
古法不可能 否——写路径安全是经典安全工程;但用 agent 快速产出 + 主循环逐行复审安全核心 + 真攻击测试闭环,是 AI-native 的实现+验证节奏。

5 · 规则候选

  • 无感分发/自更等『安全软件分发』子系统,实现前必须先出专项安全 mini-ADR 并过多轮独立对抗安全评审;评审通过前不得 code——防『安全感而非安全性』(给未验证工件盖『已验证』章比不验更危险)。
  • 供应链签名优先 keyless OIDC attestation(无长期私钥可窃)而非长期裸私钥;消费端离线验签须绑死 OIDC subject 三元组(repo+workflow+ref)+ issuer 精确 + artifact digest 绑定 + fail-closed 无 fallback + 内置 sigstore trusted root。
  • 进程/daemon 对用户 dotfile 的净新自动写路径 = RCE 面,须单独威胁建模 + 写路径安全(tar typeflag 白名单拒软链 entry / 父链逐级 EvalSymlinks / O_EXCL|O_NOFOLLOW 写原语二分 / 确定性拒软链非跳过)+ 真攻击测试(断言攻击未得逞,非仅返回 error)。
