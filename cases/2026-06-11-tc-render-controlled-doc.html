4关键判断

判断一先把「样式不好看」翻译成约束与机制问题,再动一行 CSS
背景
最初诉求只有一句「附件模版样式风格不太好」。直接重画 CSS 是最顺手的路径;但分析发现真正根因是三个非视觉问题:esc() 把内容压平成墙文、480px 内联 iframe 首屏被页头吃掉四成、四类文档共用一种阅读模式。
选项
直接改配色字体出几版皮肤 ／ 先量渲染环境(multica 前端 480px sandbox iframe)与内容结构,把问题拆成「内容结构×首屏经济学×类型差分」再设计
选择
后者。读前端嵌入代码拿到真实视口,把「按情况用不同模版」具体化为「同一外壳、按类型换首屏」,设计与字段契约一起定。
事后看 · 核心收获
对。最终 diff 里 CSS 只是配角——anyOf 双形态字段契约才是观感翻身的主力;当初直接调色的话,墙文和首屏浪费会原样保留。
古法不可能
一轮内同时读完渲染管线、前端嵌入实现、全部历史文档实例并交叉出三个根因,再用同源真实数据出八版可点开的原型——人工做这轮侦察加打样要按周计。
判断二测试禁用集用「区段封闭减白名单」,不用采样枚举——被评审实证穿透一次就立法
背景
emoji 禁集第一版测试只枚举 14 个常见码点。代码评审者构造 ✔U+2714/✖U+2716/☑U+2611/❤/⚡ 注入,全部穿过断言——而 ✔✖☑ 恰是「对勾替身」最可能的回归形态。
选项
往枚举里补几个漏网码点 ／ 整区段封闭(U+2600–U+27BF ∪ U+1F000–U+1FFFF)减白名单(✓/☐)
选择
整区段减白名单。封闭口径其实在计划评审时已立法(v4 标准 3②),实现时打了折扣,代码评审把它拉回。
事后看 · 核心收获
对,且形成闭环证据:计划评审轮把「无 emoji」从开放列表钉成封闭码点集,代码评审才有依据打回采样实现——两道门各抓了一半。
古法不可能
评审者对现产物做全码点扫描并构造五个替身字符实证穿透,分钟级;人工评审通常瞥见「有 emoji 断言」即放行。
判断三活体验证自带 --no-transition——发布动作与状态机解耦的护栏在计划层就埋好
背景
验收要求向真实 issue 发布一篇 case 看内联渲染;但 publish 默认触发入口转换,会把在建的 plan issue 拍进「复盘-待审+评审中」,且按本团队不变量巡检必产生违规告警。
选项
找沙盒 issue ／ 直接发本 issue,接受污染后手工摘标签 ／ --no-transition 显式跳过转换 + 发本 plan issue
选择
第三种。这是计划 v1 评审 B4 抓出的雷,v4 把它写进完成标准 5 的命令行里。
事后看 · 核心收获
对。发布后 TEA-103 仍是「计划-已批准 · 实现中」,全程零污染零手工补救;逃生口旗标第一次在真实验收里兑现价值。
古法不可能
判断本身不依赖 AI,但「入口转换会引爆哪条不变量」的推演来自评审子 agent 对转换脚本与巡检脚本的穷举交叉。
判断四原型可以写意,渲染器每个格子必须有源可推——把评审 BL-1 当立法时刻
背景
定稿原型里受控条编号、审批栏「所属项目/父项计划」、研究要点框都是编辑性文案;渲染器只有 fields.json 可用。v2 复审把这暴露为 blocking:标准、测试、原型三方口径分叉,第三方按字面验收必挂。
选项
实现时现场发明(查询补数据/塞 UUID/越界加字段) ／ 计划层给每格钉死「数据源|固定文案|缺省降级」三元组,无源格降级「—」或不渲染
选择
后者。D5 逐格表成为 build 的逐项映射,豁免清单收为封闭三项。
事后看 · 核心收获
对。build 阶段四个渲染器一次写对零返工——每个「这格显示什么」的问题在计划文本里都有答案;代码评审逐格对照也因此可执行。
古法不可能
三轮计划评审以格子粒度逐项对照原型与 schema 并产出可执行立法,每轮十分钟级;人工评审不会做到格子级。

5规则候选

候选规则状态
改「文档/消息模版」类需求时,先量出真实观看视口与渲染管线(嵌入尺寸/转义路径/明暗环境),再动样式——观感问题的根因常在内容结构与首屏经济学,不在配色。待提级
测试里的「禁用集」一律区段封闭减白名单,不做采样枚举——采样断言对同义替身(✔ 替 ✓、✖ 替 ×)天然失明。待提级
渲染器/模版的每个输出格子在计划层必须钉「数据源|固定文案|缺省降级」三元组——原型是写意稿,无源格子是实现期返工与验收扯皮的最大来源。待提级
