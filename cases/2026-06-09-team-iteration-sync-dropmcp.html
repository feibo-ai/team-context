4 · 关键判断

判断:命门A 成功信号纠正:attachments 非空,而非 url 前缀
背景 plan v1.3 称「不带 issue_id → url=/uploads/users/、静默不绑」,把 url 形态当成功判据。
选项 信 plan 的 url-shape 判据 / 自己端到端实测复核
选择 实测复核:workspace mul_ token 下带不带 issue_id,upload url 都是 /uploads/workspaces/;真正的失败模式是评论 attachments 为空。PUBLISH.md 自检改以 attachments 非空为准,issue_id 仍照带(绑 issue 关联+匹配 CLI UploadFile)。
事后看 对的。url-shape 判据会在别的 token 类型下误报/漏报;attachments 非空是稳健的真信号。源码(publishDoc)+实测双印证才敢推翻 plan 的「已实测」。
古法不可能 是。能在几分钟内对真服务端发→查→撤、并交叉读 Go/TS 源码三方对账,古法(纯人工读 plan 照做)只会把误判一路带进发布。
判断:发评论 JSON body:python json.dumps 严格转义,弃 jq -n --arg 拼 shell 换行
背景 content 里 caption 与 !file 间要空行(\n\n)。第一版 PUBLISH.md 用 jq -n --arg c "$cap"$'\n\n'... 拼。
选项 jq -n --arg(shell 拼真实换行)/ python json.dumps(程序内转义)
选择 实测 jq 路径产出 body 含裸换行(JSON 规范禁止),靠 server 宽容才没报错,且发布偶发 SSL 抖动。改 python json.dumps 严格转义 + --data-binary + --retry。
事后看 对的。不该把正确性押在 server 对非法 JSON 的宽容上;python json.dumps 与 MCP 的 JSON.stringify 行为一致,稳。
古法不可能 否。这条是经典工程判断,AI 只是更快地通过实测把它逼出来。
判断:软链脏态:12 个 tc-* 全清,而非只清本期改的 4 个
背景 ~/.claude/skills 下 12 个 tc-* 都是「物理 SKILL.md+自指子软链」脏态;本期只改 4 个。
选项 只清 4 个(最小范围)/ 全清 12 个(彻底)
选择 全清。plan 决策5b 立「整目录软链→team-context/skills」单一真相;cp -R 物理副本到 repo 外备份后 rm-rf 建顶层软链;只清 4 个会留半脏态。
事后看 对的。半清是更坏的 smell,且转软链是纯基建(不改 skill 内容,迭代2 改 tc-1-start 内容反而更顺);备份+源在 git 双保险可回滚。
古法不可能 是。能一次性对 12 个目录做「备份→校验字节→删→重建→冒烟」的机械批处理且当场用 skill-list 刷新验证,古法手工逐个极易漏。
判断:命门B:改 UploadFileWithURL 签名加 issueID,而非新增方法
背景 --inline 要 upload 既绑 issue(issue_id)又返回 url;现有 UploadFile 绑定但不返 url,UploadFileWithURL 返 url 但不绑定。
选项 改 UploadFileWithURL 签名(+issueID)/ 新增 UploadFileWithURLForIssue 方法
选择 改签名。只有 1 个真调用方(agent avatar,传"")+ 5 处 test;复用其 url 返回,mini-publishDoc 仅 ~35 行,远小于复制一份新方法。
事后看 对的。~35 行落在 plan「~40-70 行」估内;新方法会重复整段 multipart+解码逻辑。先 grep 全调用方(含 _test.go)一次更新,build/test 一次过。
古法不可能 否。这是常规重构判断;AI 的价值在一次性把 def+1 真调用+5 测试点改齐且即时 go test 验证。
