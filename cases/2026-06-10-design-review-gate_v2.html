4 · 关键判断

判断:评审打回的四条 blocking 全是『新边接进旧机』的耦合雷,不是新边本身
背景 设计门两条转换边本身很简单;plan v1 把它们当独立增量来写,评审者却在既有系统里找到了四处会被新边引爆的旧约定:不变量 #4 的值域、双用途常量、复审路径、registry 分发链。
选项 按 v1 直接实现,出问题再修 / 采纳全部四条,先改 plan 再动代码
选择 全采纳。两条有线上同型先例佐证(B3=#6 的 TEA-79 病型、B4=registry 回写当日实证),另两条(B1 不变量矛盾、B2 常量耦合)是可当场证伪的结构矛盾——比先例更硬;且都是 plan 文本层小修,半天 appetite 不受影响。
事后看 对。最值钱的是 B2:若 设计-待审 进了 PLAN_PENDING_LABELS,plan-approve 的幂等补救(transition.py 明文卖点)会变成摧毁设计门的破坏性操作——这种『卖点变凶器』的耦合,作者视角几乎不可能自查出来,独立评审是唯一防线。
古法不可能 评审者对 transition.py 全部调用点做了穷举交叉(双用途常量的两个引用点、invariants 的门内门外),并以反例注入验证测试真锚,v1→v2 全程 10 分 27 秒(评论时间戳可锚定)——人工评审给不起这个密度。
判断:不变量 #4 开 carve-out,而不是给设计评审另发明一个 status
背景 设计评审中 = 已批准+in_review,直接违反 #4。三个出路:carve-out、设计评审不动 status(只挂 label)、或为设计态引入新 status 值。
选项 carve-out(#4 在 设计-待审 在场时放行 in_review) / 设计评审不动 status,只用 label / 请求 multica 加新 status
选择 carve-out。不动 status 会让『评审中』在看板/status 查询里隐身,违背双轨语义(label 精确、status 粗轨);新 status 动到 CLI/服务端,超出 out-of-scope 红线。carve-out 是最小的诚实:状态机承认『已批准的东西可以回到评审中』这一真实语义。
事后看 对,且 live 验证干净:TEA-99 处于 carve-out 态时全量 strict 扫描 0 违规、离开后亦 0;评审者的反例(已批准+in_review 但无 设计-待审)由代码路径直接可证仍被 #4 抓住(ok4 仅在 设计-待审 在场时含 in_review)——豁免没有变成漏洞。
古法不可能 carve-out 的正确性靠『豁免态在场/不在场 × 93 issue 全量』的即时对照实验确认,分钟级;古法只能拍脑袋上线再等漂移。
判断:『从未请审就开工』的盲区选择如实声明,而不是造一个会误报的不变量
背景 brief 建议巡检对跳过设计评审给软提示;但 issue 数据无 layer 字段,巡检无法区分『项目层违纪跳过』与『任务层合法跳过』。
选项 照 brief 做巡检提示(对全部任务层 build 持续误报) / 放弃机制检查,纯靠纪律 / 拆开:能测的(评审中开工/评审挂起)做进机制,不能测的(从未请审)如实声明盲区并落 pre-flight 纪律
选择 第三种。误报会让巡检报告失去信任(狼来了效应),比盲区更伤;盲区写明在 tc-design-review skill 里,并记 issue.metadata 为将来承载 layer 的现成落点。
事后看 对。诚实的盲区声明 + 两个能测的边界(build-start 告警、staleness)覆盖了大部分风险面;剩余靠 tc-4-build pre-flight checklist,与『计划评审项目层必走』同一执法等级。
古法不可能 判断本身不依赖 AI;但『误报面=全部任务层』这个量化依据来自对 93 个 issue 数据形状的即时核查(metadata 全空),不是猜的。

5 · 规则候选

  • [ ] needs DRI promotion decision — 给既有状态机/规则系统加新边时,评审清单必须含『新边 × 每条旧不变量 × 每个共享常量调用点』的穷举交叉——耦合雷全在旧约定里,不在新边本身。
