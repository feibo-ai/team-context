复盘时核对 plan 完成判据的方法——逐条 met/not 标注、可观测信号与未达成原因的写法。

# 前置

- 拿到被复盘的 plan（项目层 plan 文档 HTML / 任务层 3 句 mini-plan），完成判据以 plan **原文**为准，不凭记忆复述。
- 即使用户说"判据都满足了"，也要逐条核对——这是复盘的入场核验，不是走过场。

# 逐条核对流程

对 plan 里的每一条完成判据：

1. **原样引用判据文本**（不改写——防止对着事后放宽的判据打勾）。
2. **标注 met / not met**：
   - `[x]` 达成：附**可观测信号**——测试全绿、可点开的链接、数据、截图路径等能被第三人核验的证据，不是"我觉得做完了"。
   - `[ ]` 未达成：写明确原因（scope 变了 / 被 blocker 挡住 / 主动砍掉…），不许留空。
3. **给结论**：
   - 全部 met → 结论用「标准全数达成」的定式表述。
   - 存在 not met → 结论如实写部分达成，并确认每个未达成项是否需要新开 issue 跟进（不要让它无声消失）。

# 与发布的关系

fields.json 的 `criteriaResults` 每条都必须带 met 标注，met 的带信号、not met 的带原因——publish 校验不过会拒发。字段 schema 以 tc-render skill 的 references/publish-contract.md 为准。
