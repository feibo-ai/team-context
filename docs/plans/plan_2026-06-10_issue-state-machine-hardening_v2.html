方案

【背景/根因】去本地MCP 迁移只迁了文档发布这一半:publish.py 用 exit-1 硬校验收口了文档质量边,但原 MCP 工具(plan_approve/case_review/project_kickoff)内置的原子状态转换被拆成 skill 散文里的 bash 块。结果:7 个状态只有 2 个有指令;计划-草稿/复盘-待审/研究 三个入口 label 无主(labels.md 声称 publish.py 会打,实际代码不打);tc-5-review 收尾块写『case 经第二 session 评审通过后』执行,但评审 session 不加载该 skill、原 session 已结束,无人执行;且 skill 里的 `label add <issue> <名称>` 命令本身跑不通——CLI 实际签名是 `<issue-id> <label-id>`,只收 label UUID(v1 评审期实测确认)。线上全量 91 个 issue(TEA-7~97,分页拉全)实测:TEA-95/TEA-70 复盘-已审 却 status todo、父 plan 未连带关闭(收尾散文块只执行了一半的实证);复盘-待审 滞留 9 个;已升级+已批准 并存 4 个;tc-friday 查询(--status done --label 复盘-待审)按规范时序永远空集(线上当下命中的 TEA-62/33 恰是漂移产物);daily-kickoff『待启动』桶(计划-已批准+todo)数学上不可达,实测 0 条。 【设计核心一:转换收口进脚本(仿命门B)】transition.py 与 publish.py 同目录共库。每个子命令 = 一次原子转换:前置态校验(意外现场只警告不阻断,保证存量可修)→ 按序执行 label 增删 + status 置位 + 父链连带 → issue get 复核后置状态,不符 exit 1(P-7 真验证内建)。label 引用一律名称,经 `multica label list --output json` 运行时解析为 UUID;禁止硬编码 UUID(UUID 是 workspace 数据,labels.md 才是单源,create-labels.sh 重建后 UUID 会变);某 label 在 workspace 不存在时 exit 1 并提示跑 scripts/create-labels.sh,绝不静默跳过。幂等:执行前 issue get 预读当前 labels/status,『已存在/已是目标值』直接视为成功跳过,不解析 duplicate-add 报错文案(CLI 报错串无契约保证)。并发语义:last-writer-wins,后置复核会以 exit 1 暴露被插队,docstring 注明。--dry-run 只打印将执行的 CLI 序列。 【设计核心二:执行者绑定——子 agent 编排】团队成员人手大量 agent:凡 SOP 说『第二个 session』(plan 评审、case 评审、测试验证),由子 agent 承担——子 agent 天然全新上下文,满足独立性要求且无需人工换 session;verdict 返回到编排 session 的那一刻就是转换执行点(plan-approve / case-finalize)。评审与状态转换在同一编排回合内闭合,把『等一个不存在的第二 session 来收尾』的无主窗口收窄至单回合;编排 session 在 verdict 后、转换前死亡的残留态(评审中/待审 长期滞留),由 issue_invariants.py 的 48h staleness 警告兜底捕获。转换权始终归编排 session,子 agent 只产出 verdict、不碰状态。本 plan 的 TEA-97 评审流程即此模式的首次实战(v1 评审子 agent 已完成一轮 changes-requested)。 【语义修正:批准 ≠ 开工】plan-approve → 加 计划-已批准、摘 {计划-草稿, 计划-评审中, 计划-已升级}(摘已升级是 v1 评审 B2:不摘则 upgrade→re-approve 后 TEA-79 型并存漂移周期性再生)、status todo(待启动)。tc-4-build 首个 build session 在 pre-flight 通过后执行 build-start → in_progress(与开工卡同时机)。plan-request-review → 加 计划-评审中 + status in_review(tc-3-plan 与 tc-1-start Step4 都显式调用)。research 发布(findings 非空)即 status done(研究产物=发现已交付)。DRI 拍板 kill → cancel:status cancelled + 摘除精确集合 {计划-草稿, 计划-评审中, 计划-已批准, 计划-已升级, 复盘-待审, 复盘-已审}(不含 研究/古法不可能/投注表/代码评审/倦怠预警)。plan 升级 → 摘 计划-已批准、加 计划-已升级+计划-草稿、status 回 todo,再走 request-review。case-finalize → 加 复盘-已审/摘 复盘-待审 + status done + 父 plan done 并清父未决流程 label(草稿/评审中/已升级,保留已批准)+ 祖父 research 尽力关闭;--keep-parent 跳过父链操作(phase case 专用:如 TEA-62 是 actionow-hardening 的 phase-a 复盘,不得连带关掉仍在进行的 TEA-57)。 【修订面清单】skills:tc-1-start(Step4 请审/Step5 批准与 kill 路径换 transition.py、Step2 注明 research 自动转换)、tc-2-research(发布即 研究+done 说明)、tc-3-plan(批准块换 session)、tc-friday(盘点查询改 --label 复盘-待审,去掉 --status done)。standards/labels.md:① 每行执行者列精确到 transition.py/publish.py/remote-MCP(投注表/倦怠预警/代码评审 仍归 remote MCP;注明 plan_request_review 的 label 写路径废弃,统一走 #f97316 vs 文档 #3B82F6),注明 2026-06-10 以线上为准核对;⑤ 更新 last reviewed。tc-ops:新增 issue_invariants.py(分页硬要求)+ SKILL.md 注册。tests:新增 test_transition.py(mock CLI runner:决策逻辑/幂等/keep-parent/解析失败降级)+ publish.py 入口转换与 exit 2 契约测试 + test_contract_probe.py 增 issue get/label list JSON 形状探针(防 CLI 升版静默漂移,本机 v0.4.15)。autopilots 查询零改动(语义修正后自然成立)。最后按 doc-drift 规则全仓扫尾旧引用(cases/、decisions/、历史 plans 豁免)。 【backfill
