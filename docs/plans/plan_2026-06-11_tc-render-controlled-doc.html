方案

【设计输入（已定稿·不在本 plan 重审）】视觉方向经两轮原型评比定稿为「受控文档」：~/feibo/tpl-prototypes/ 中 i-shenpi.html（复盘）/j-shenpi-plan.html（计划）/k-shenpi-research.html（研究）/l-shenpi-handoff.html（交接）四件套+index.html 对比画廊；设计三原则=全中文、端正无倾斜、重点突出（要点/结论前置进 480px 内联首屏，视口依据=multica html-attachment-preview 的 480px 内联 iframe）。本 plan 只管把定稿落进 publish.py，不重开视觉讨论。【改动面】全部落在 team-context 仓 skills/tc-render/：①assets/style.css 1775B→约 6KB，新增受控文档类（sheet/ctrl/appr/sealcell/sq/tally/keypoints/clause/formal 表/judge/hind/next/gate/xbox），类别主色以 CSS 变量 --type 按类型注入；②publish.py shell() 重构为受控文档骨架（框线纸面+受控条+审批栏+标题区），四个 render_<type> 各自装配类型首屏：计划=完成标准数+拍板要点+预算反白格；研究=问题黑框+总体裁决格+裁决要点；复盘=结论格(criteriaResults met n/m)+要点提示(取 keyJudgments[].title)；交接=接收/唯一输入/禁区计数+第 1 节即下一步；③SCHEMAS 扩展，全部 optional、additionalProperties 仍 false。【字段契约决策】D1 双形态正文：正文字段收 string|string[]，数组项逐条 esc() 后渲染为段落/编号条款，string 走旧路径——化解 esc() 压平墙文的最小改动，不引入 markdown 解析（安全面不变）。D2 plan 新增 keyDecisions[]/risks[]（可选）：首屏「拍板要点」与方案段 D/R 条款的数据源；缺省时拍板要点降级为完成标准前 3 条。D3 复盘要点提示不新增字段：直接取 keyJudgments[].title。D4 审批栏：计划=拟制(dri)/评审(reviewer)/批准(固定「待负责人拍板」)/日期；复盘=拟制(固定 tc-5-review session)/评审/批准/日期；研究/交接用固定中文文案+现有字段，不为表头新增必填。【渲染安全不变量】全字段 html.escape；零外链零 JS 零 emoji 零 rotate；产物以 <!DOCTYPE html 起头（MIME 守卫不动）；safe_out_path/命门B 发布链路/transition 调用一概不碰。【测试】既有套件保绿（向后兼容锚）；新增：四类结构锚点断言、双形态渲染断言（array→多段落）、负面断言三连（英文标签/emoji/rotate 零命中）、旧版纯 string fields.json 回归用例。【边界 out of scope】不改 transition.py；不改命门B/灾备契约语义（PUBLISH.md 仅更新字段段）；不动 multica 前端；暗色变体与第一/二批其他原型方向（卷宗/台账/学报/工单/红头/蓝皮书/内参）归档不实现；四个消费 skill 的 fields 凑法提示属 build 交付清单内一次性小改，不扩大流程。【风险】R1 校验扩展写错会挡现网发布——以旧 fields 回归用例为硬锚，全部新能力 optional；R2 CSS 体积约增 3 倍（仍 <8KB，srcDoc 内联零外链请求，可忽略）；R3 字体栈 Songti/PingFang/STZhongsong 为 macOS 最优、非 Mac 退化系统衬线（团队全员 Mac，保留 fallback）；R4 活体验证在真实 issue 留一条评论（选本 plan issue 自身，append-only 可接受）。【评审记录】v1 送审：pending。
