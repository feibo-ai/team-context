1目标

把 SOP 工具链(tc-* skill + 发 HTML 受控文档 + label 状态机)的真实采用,从『仅 owner 曾振华一人』扩到全队 5 人。办法是适配团队真实环境——100% 本地、一人一机、多工具(Claude Code + Codex + Hermes + Opencode 混用)——让每种工具都能调用工具无关的 tc-render 脚本,并按项目层/任务层分层,使高频建 issue 的成员(荣灿约 36 个/天)不被全流程拖垮。

5方案要点

【背景与定位】2026-06-24 对全工作区 998 个 issue 做了全量普查,定位:tc-render / RPI / label 状态机这整套 SOP 工具链,真实采用者只有 owner 曾振华一人。三组证据互相印证——① 走 skill 流程合规率:曾振华 92% · 荣灿 36% · qiuhaoqi ~1%;② 发 HTML 受控文档:全工作区仅 67 个 issue 发过,其中 60 个是曾振华,荣灿 3/755、qiuhaoqi 1/101;③ 挂流程 label:曾振华 51 个,其余全员合计 17 个。与此同时约 500 个 issue 把 Markdown 长文直接塞进 description(荣灿一人 444),即把 multica 当裸 issue tracker 用。

【根因·最上游】15 个 tc-* skill 已上传 multica workspace,但 6 个助理 agent 全部 0 skill、instructions 是同一份 2366 字模板且零 SOP 引用;工具链只活在曾振华本地 ~/.claude/skills/(他跑本地 Claude Code)。更关键的结构性事实:团队 100% 本地、一人一机(15 个 runtime 分布在 5 台机器),且多工具混用——Codex 6 个(最多)、Claude 5、Hermes 3、Opencode 1,每个人都同时装 Claude 和 Codex。而 SOP 是 Claude Code skill 格式,在 Codex / Hermes / Opencode 里不自动加载。结论:用非 Claude 工具干活的人(尤其荣灿用 Opencode/Codex)物理上够不到 SOP——这是结构性锁死,不是不自律。

【为何不换轨/不砍 label】有人会问能不能用 multica 内置 status 替代自定义 label。不能:status 是固定 7 枚举(backlog/todo/in_progress/in_review/done/blocked/cancelled)、不可自定义、不能按项目定义工作流,而团队 RPI 有 8 个流程子状态要表达,in_review 一格背 plan评审/设计评审/case待审三义、todo 一格背 草稿/已批准/已升级/设计已审四义,7 个格子物理装不下。所以『label vs status』是伪命题,换轨救不了一列没人坐的车。砍掉 0 使用的死 label(投注表/倦怠预警/代码评审/古法不可能/计划-已升级)是可做的瘦身,但属独立小事,不是本期目标。

【路径选择】不碰 cloud(团队没有任何 cloud agent,全是本地 runtime)。走『本地 + 跨工具可移植 + 分层』。好消息是地基已经对了:tc-render 的 publish.py 与 transition.py 是纯 Python、工具无关,任何 agent(包括 Codex)都能调;团队 CLAUDE.md 也早写了『Codex/其他 agent 直读该目录照做』。缺的只是把这个工具无关底座真正接到每个人的每种工具上——最后一公里。

【分层判定规则(DRI 已拍板)】项目层 vs 任务层是本方案承重墙,采用『多触发器任一』:issue 满足任一即项目层——① 写/改代码或配置且会合进 main;② 需跨 session 或跨人交接;③ 对外或影响他人产出;④ 预计耗时 > 半天;否则任务层。项目层必走全流程(plan HTML 文档 + 评审门 + label 状态机),任务层放行轻量(3 句 mini-plan 自批即可)。四触发器是『人该怎么做』的规范;其中 ②跨交接/③影响他人/④>半天 无法从 issue 字段机读,故度量侧改用一个 recall 导向的机读 project-ish proxy(标题前缀 ∨ description>600字 ∨ 含 Markdown 标题 ∨ 关联 GitHub PR/repo 链接;已验 issue list 返回 description)作为可计算近似,专供 nudge 命中与覆盖率分母(完成标准 5)用——它过近似四触发器、宁多抓不漏。

【四步动作】(1) 下发:每台机器各人自跑 multica skill pull(owner 只验收不代跑),把 15 个 tc-* skill + 脚本拉到本地,5 人全覆盖并核 CLI ≥v0.4.13。(2) 架桥:给非 Claude 工具配可用入口——Codex 用 AGENTS.md、Opencode 用其配置,指向 team-context 脚本 + 一页《发文档/流转状态怎么调 publish.py/transition.py》工具无关速查;本期验收=被指到速查后能在 Codex/Opencode 各跑通一个 HTML 文档(连通性),真·自发触发列后续探索(B2)。(3) 分层:把已拍板的判定规则落 standards/ 与 CLAUDE.md,含机读 proxy 的可执行实现。(4) nudge:扩 issue_invariants 盲区档命中为机读 proxy,命中且未采用者经 notify_team 在 daily 群卡新增 nudge 段 @创建者(守透明文化单通道、不复活 P2P;前置补 team_members roster);覆盖率分子分母全字段可算,首周基线 4%。

【本期不处理 agent 侧绑定(N3-plan)】根因里『6 个助理 agent 全 0 skill』属实,但本期不做 agent 侧 skill 绑定/instructions 改写——这些 agent 全是 idle 的 local agent,团队当前直接用各自工具干活、几乎不经 multica agent 派活(全工作区仅 4 个 agent-created issue)。agent 侧绑定留作后续(若团队转向 agent 派活模式再做),本期聚焦人/工具直接可达这条已验证有效的路径。

【排序与验证】先做 1+2,并在荣灿的 Codex/Opencode 环境上验证『提示后能跑通』(他工具最杂、当前最不合规,是最强的可移植性样本);跑通再铺 3+4。1 周 appetite,边界:本期只做布线/文档/一个 nudge,不重写状态机、不动 label 设计、不碰 cloud、不做 agent 侧绑定、不复活 P2P。

【评审迭代】计划评审门①:v1 经独立 staff engineer 评审 changes-requested(B1 架桥验收/B2 nudge 抓不到人/B3 分层规则缺位),v2/v3 逐条修订并经两位独立 reviewer 复核至 approved(机读 proxy 分母 + 基线 4% + 触发防过拟合)。设计评审门②:v3 设计载体经独立 staff engineer 评审 changes-requested——B1『push 创建者本人』撞团队 notify_team 单通道(P2P 已按透明文化移除)、B2『自发触发』在静态 AGENTS.md 下机制未证可达。v4 据 DRI 决策收口:B1→notify_team 群卡 @创建者 + 补 roster + 挂 daily;B2→本期验收降级为『提示后能跑通』、自发触发列后续;并吸收 N1(proxy 落点=扩 issue_invariants.py、已验 list 返回 description)、N2(skill pull 各人自跑)、N3(挂 daily 非月度)。
