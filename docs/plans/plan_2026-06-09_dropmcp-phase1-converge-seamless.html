目标

Goal
在不拆库、不引入新服务端前提下完成『去本地MCP』Phase 1(DRI 拍板:全纳入本期):堵死 5-agent 评审揪出的 7 类发布漏洞(校验可绕 / fields 静默吞 / registry 分发有损 / 版本散落无 preflight / --out 越权写 / 裸 traceback / 双端不对称),并复用既有 multica daemon 把 skill·md 分发更新与版本·健康采集做成自动、静默、用户无感 + 零配置安装。

完成标准

  • [P1·校验] 全仓四类文档发布无旁路:删 tc-handoff『快捷 markdown』绕过口后 grep 无任何跳过 publish.py 硬校验的入口;fields 拼错 key 或漏 required → publish.py exit 1(JSON Schema additionalProperties:false + required);case 质量闸折叠空白后计数,纯空格/重复占位过不了 ≥100 字符。
  • [P1·安全] red-team 四项各有『失败→通过』用例:--out ../../x 越权写被白名单拒(exit 1);坏 JSON/缺 config 打友好 ERROR 不泄露裸 traceback;token 不进 argv(ps 抓不到);--inline 非 text/html MIME 被拒。
  • [P1·分发] multica skill pull 后 SKILL.md 仍带 owner(buildSkillMd 补 owner,cmd_skill.go:259),multica skill lint 零 owner(SOP ❌5)警告;CI repo→registry 对账不一致时 fail。
  • [P1·版本] multica doctor 把散落版本门槛(v0.4.11/12/13)收成一个发布前硬门:CLI/skill 版本不达标时 exit 非 0。
  • [P1·对称] Codex 与 Claude 走同一可执行入口(两端 Bash 调同一命令)+ 生成 Codex skills-index;命门契约写成 publish-contract-v1.yaml + CI 探针对齐前端权威正则 file-cards.ts:NEW_FILE_CARD_RE。
  • [P1·无感分发] 用户零手动命令即拿到最新 skill/md:daemon 起 skill 同步循环(仿 workspaceSyncLoop),本地落后 registry 时静默拉齐;一次真跑——改一 skill 推 registry、不手动 pull,数分钟内本地更新到位且 lint 通过。
  • [P1·自动采集] daemon 心跳带结构化遥测(CLI 版本 + skill 版本/健康 + 命门成功率),服务端可见『谁在什么版本』;隐私边界明文:采版本/健康/命门成功率,不采文档内容/token/源码。
  • [P1·零配置] install.sh 补 token/workspace 自动发现,新成员 onboarding 压到一条命令即可用(零手动编辑 config.json / MCP),首登即配齐。

方案

【背景】本 plan 落实两份 ADR:2026-06-08-drop-local-mcp(去本地MCP,RPI 文档流迁到 skills+CLI+命门A,迭代2 删整包)与 2026-06-09-rpi-publish-architecture-redeliberation(5-agent 评审 + 3 架构师再审议,三家收敛=现在做不拆库轻量收敛=Phase 1,核心下沉留 Phase 2 条件触发)。北极星(DRI 追加):极致简化安装+使用,让分发/更新/采集自动、静默、用户无感。本 plan 把 Phase 1 从『修评审漏』升级为『修漏 + 无感地基』双线,DRI 拍板『所有』全纳入本期。 【张力的化解 · 本 plan 核心架构判断】交接担心『无感自更+采集需常驻组件=Phase 2 territory,又与极致简化安装冲突』。前置研究(读 tc-multica/server)证明:multica daemon 已存在且已做最难的部分——已每 6h 自动拉 GitHub release 自更 CLI 二进制(auto_update.go:42,带 claim barrier 不打断在跑任务)、已每 1m 采集 CLI transcript 用量并 POST 回服务端(ambient_usage.go:128)、心跳已带 X-Client-Version 头且响应已能下发 PendingLocalSkills 并回传 skill inventory。故『复用 daemon 做 skill/md 无感自更 + 版本/健康采集』不是把重 Phase 2 搬进来(无新 server、不拆库),而是沿 daemon 已有的槽延伸;daemon 是成员已在跑的既有组件(autopilot 依赖它常驻),不增安装负担。结论:复用既有 daemon,北极星不触发 Phase 2。 【Phase 1 评审漏七条 · 经北极星重读】① 收口单命门:publish.py 内部 exec multica issue comment add --inline,删 PUBLISH.md §2 手跑兜底,命门A 降契约化灾备(publish-contract-v1.yaml 对齐前端权威正则 file-cards.ts:NEW_FILE_CARD_RE + CI 探针);② fields JSON Schema(additionalProperties:false + required),阈值从 schema minLength/minItems 派生(消散落魔数),计数前折叠空白 + 类型断言(堵质量闸空格绕过与类型混淆);③ 删 tc-handoff『快捷 markdown』绕过口,四类文档无旁路、校验回强制;④ 修分发:buildSkillMd 补 owner(cmd_skill.go:259 漏写致 pull 后自伤 lint;Phase 1 唯一越过纯 skill 侧的 Go 改)+ sync-team-config 改 create-or-update + 推 files[] + CI repo→registry 对账,registry 降派生只读投影、开发机走 git 软链;⑤ multica doctor preflight 收散落版本门槛成一发布前硬门;⑥ Codex 对称:收口后两端 Bash 调同一命令 + 生成 Codex skills-index;⑦ red-team 致命四项:--out/slug 路径白名单+仓内断言、裸 traceback→友好 ERROR+exit 1、token 移出 argv、--inline MIME 校验。其中 ④⑤ 即无感升级口(④让分发可被 daemon 静默拉、⑤为无感自更的版本门)。 【无感地基 · 本期同交付 · 复用 daemon】(DRI 拍板『所有』,不切 1.5)(a) daemon skill/md 同步循环:仿 workspaceSyncLoop 加 skill-sync ticker,本地落后 registry 时静默拉齐(复用既有
