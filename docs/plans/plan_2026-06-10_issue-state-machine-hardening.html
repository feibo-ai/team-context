方案

【背景/根因】去本地MCP 迁移只迁了文档发布这一半:publish.py 用 exit-1 硬校验收口了文档质量边,但原 MCP 工具(plan_approve/case_review/project_kickoff)内置的原子状态转换被拆成 skill 散文里的 bash 块。结果:7 个状态只有 2 个有指令;计划-草稿/复盘-待审/研究 三个入口 label 无主(labels.md 声称 publish.py 会打,实际代码不打);tc-5-review 收尾块写『case 经第二 session 评审通过后』执行,但评审 session 不加载该 skill、原 session 已结束,无人执行。线上 50 个 issue 实测:TEA-95/TEA-70 复盘-已审 却 status todo、父 plan 未连带关闭;TEA-85/90/91 零 label;tc-friday 查询(--status done --label 复盘-待审)按规范时序永远空集;daily-kickoff『待启动』桶(计划-已批准+todo)数学上不可达——批准即 in_progress 把 todo 态跳过了。 【设计核心一:转换收口进脚本(仿命门B)】transition.py 与 publish.py 同目录共库。每个子命令 = 一次原子转换:前置态校验(意外现场只警告不阻断,保证存量可修)→ 按序执行 label 增删 + status 置位 + 父链连带 → issue get 复核后置状态,不符 exit 1(P-7 真验证内建)。幂等:label 已存在/已是目标 status 视为成功。--dry-run 只打印将执行的 CLI 序列。publish.py 发布成功后 import 同库函数做入口转换,失败时明确报『评论已发、转换未完成』+ 补救命令,绝不静默。 【设计核心二:执行者绑定——子 agent 编排】团队成员人手大量 agent:凡 SOP 说『第二个 session』(plan 评审、case 评审、测试验证),由子 agent 承担——子 agent 天然全新上下文,满足独立性要求且无需人工换 session;verdict 返回到编排 session 的那一刻就是转换执行点(plan-approve / case-finalize)。评审与状态转换在同一编排回合内闭合,结构性消灭『等一个不存在的第二 session 来收尾』的无主窗口。转换权始终归编排 session,子 agent 只产出 verdict、不碰状态。 【语义修正:批准 ≠ 开工】plan-approve → status todo(待启动);tc-4-build 首个 build session 在 pre-flight 通过后执行 build-start → in_progress(与开工卡同时机)。plan-request-review → 加 计划-评审中 + status in_review。research 发布(findings 非空)即 status done(研究产物=发现已交付)。DRI 拍板 kill → cancel 子命令(status cancelled + 清全部流程 label)。plan 升级 → 摘 计划-已批准、加 计划-已升级+计划-草稿、status 回 todo,再走 request-review。case-finalize 连带:case done + 加 复盘-已审/摘 复盘-待审 + 父 plan done + 清父 plan 未决流程 label(草稿/评审中/已升级,保留 已批准)+ 祖父 research 尽力关闭(legacy 链兜底)。 【修订面清单】skills:tc-1-start(Step4 请审/Step5 批准与 kill 路径换 transition.py、Step2 注明 research 自动转换)、tc-2-research(发布即 研究+done 说明)、tc-3-plan(批准块换 plan-approve、评审 gate 子 agent 化)、tc-4-build(pre-flight 后加 build-start、测试验证子 agent 说明)、tc-5-review(发布自动 复盘-待审+in_review、收尾块换 case-finalize 且执行者=评审 verdict 返回后的编排 session)、tc-friday(盘点查询改 --label 复盘-待审,去掉 --status done)。standards/labels.md:加/删时机与执行者列改为 publish.py/transition.py、状态机图补 status 轨、新增 5 条不变量表、更新 last reviewed。tc-ops:新增 issue_invariants.py + SKILL.md 注册。tests:新增 test_transition.py(mock CLI runner,测决策逻辑与幂等)+ publish.py 入口转换测试。autopilots 查询零改动(语义修正后自然成立)。最后按 doc-drift 规则全仓扫尾旧引用(cases/、decisions/、历史 plans 豁免)。 【backfill 清单(实施段逐条执行并记入 PR)】TEA-95 case-finalize(连带关 TEA-93);TEA-70 case-finalize(连带关 TEA-66 并清其 草稿/评审中);TEA-62、TEA-94、TEA-54 → in_review;TEA-90、TEA-82 验证 case 评论已发布后补 复盘-待审 + in_review;TEA-91 补 研究 label;TEA-78、TEA-76 验证 findings 已发布后 → done;TEA-85 → in_progress(其 case TEA-90 已存在,工作已开展);TEA-75 清 草稿+评审中;TEA-69 清 复盘-待审;TEA-79 摘 计划-已升级(已批准成立)。测试垃圾 issue TEA-88/87/86/84 不动。全部为 status/label 操作,可逆。 【边界 out of scope】不改 multica CLI/服务端;不动 claude_md_team_global.md(红线文件,走月度 review);不发飞书广播(开工卡/收工卡由用户自行决定);不为本修复回溯补建 research issue;cases/、decisions/、已发布历史文档不回改;blocked/backlog 两个状态本期不启用(留待真实场景出现再定义)。 【风险与对策】publish.py 是命门B 收口件,改动有回归面 → 入口转换做成独立函数+独立测试+--no-transition 逃生口,主发布路径逻辑零改动;backfill 是线上变更但全部可逆,清单先经本 plan 子 agent 评审;transition.py 后置复核若因 CLI 输出格式变化误报,--dry-run 可人工核对。
