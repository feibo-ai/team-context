1目标

把 SOP 工具链(tc-* skill + 发 HTML 受控文档 + label 状态机)的真实采用,从『仅 owner 曾振华一人』扩到全队 5 人。办法是适配团队真实环境——100% 本地、一人一机、多工具(Claude Code + Codex + Hermes + Opencode 混用)——让每种工具都能调用工具无关的 tc-render 脚本,并按项目层/任务层分层,使高频建 issue 的成员(荣灿约 36 个/天)不被全流程拖垮。

5方案要点

【背景与定位】2026-06-24 对全工作区 998 个 issue 做了全量普查,定位:tc-render / RPI / label 状态机这整套 SOP 工具链,真实采用者只有 owner 曾振华一人。三组证据互相印证——① 走 skill 流程合规率:曾振华 92% · 荣灿 36% · qiuhaoqi ~1%;② 发 HTML 受控文档:全工作区仅 67 个 issue 发过,其中 60 个是曾振华,荣灿 3/755、qiuhaoqi 1/101;③ 挂流程 label:曾振华 51 个,其余全员合计 17 个。与此同时约 500 个 issue 把 Markdown 长文直接塞进 description(荣灿一人 444),即把 multica 当裸 issue tracker 用。

【根因·最上游】15 个 tc-* skill 已上传 multica workspace,但 6 个助理 agent 全部 0 skill、instructions 是同一份 2366 字模板且零 SOP 引用;工具链只活在曾振华本地 ~/.claude/skills/(他跑本地 Claude Code)。更关键的结构性事实:团队 100% 本地、一人一机(15 个 runtime 分布在 5 台机器),且多工具混用——Codex 6 个(最多)、Claude 5、Hermes 3、Opencode 1,每个人都同时装 Claude 和 Codex。而 SOP 是 Claude Code skill 格式,在 Codex / Hermes / Opencode 里不自动加载。结论:用非 Claude 工具干活的人(尤其荣灿用 Opencode/Codex)物理上够不到 SOP——这是结构性锁死,不是不自律。

【为何不换轨/不砍 label】有人会问能不能用 multica 内置 status 替代自定义 label。不能:status 是固定 7 枚举(backlog/todo/in_progress/in_review/done/blocked/cancelled)、不可自定义、不能按项目定义工作流,而团队 RPI 有 8 个流程子状态要表达,in_review 一格背 plan评审/设计评审/case待审三义、todo 一格背 草稿/已批准/已升级/设计已审四义,7 个格子物理装不下。所以『label vs status』是伪命题,换轨救不了一列没人坐的车。砍掉 0 使用的死 label(投注表/倦怠预警/代码评审/古法不可能/计划-已升级)是可做的瘦身,但属独立小事,不是本期目标。

【路径选择】不碰 cloud(团队没有任何 cloud agent,全是本地 runtime)。走『本地 + 跨工具可移植 + 分层』。好消息是地基已经对了:tc-render 的 publish.py 与 transition.py 是纯 Python、工具无关,任何 agent(包括 Codex)都能调;团队 CLAUDE.md 也早写了『Codex/其他 agent 直读该目录照做』。缺的只是把这个工具无关底座真正接到每个人的每种工具上——最后一公里。

【分层判定规则(DRI 已拍板)】项目层 vs 任务层是本方案承重墙,采用『多触发器任一』:issue 满足任一即项目层——① 写/改代码或配置且会合进 main;② 需跨 session 或跨人交接;③ 对外或影响他人产出;④ 预计耗时 > 半天;否则任务层。项目层必走全流程(plan HTML 文档 + 评审门 + label 状态机),任务层放行轻量(3 句 mini-plan 自批即可,见任务层惯例)。该规则要求可机器近似判定,既用于落档约束,也作为 nudge 覆盖率(完成标准 5)的分母口径。

【四步动作】(1) 下发:每台机器跑 multica skill pull,把 15 个 tc-* skill + 脚本拉到本地,5 人全覆盖(并核 CLI 版本 ≥v0.4.13)。(2) 架桥(关键、最不确定):给非 Claude 工具配可用入口——Codex 用 AGENTS.md、Opencode 用其配置,指向 team-context 脚本 + 一页《发文档/流转状态怎么调 publish.py/transition.py》工具无关速查,并把它做到能让 agent 在真实任务里『自发触发』(完成标准 2 验收的是触发,不是手敲)。(3) 分层:把上面已拍板的判定规则落 standards/ 与 CLAUDE.md,含机器近似判定实现。(4) nudge:把 issue_invariants 盲区档从 report-only 升级为 push 给创建者本人,并扩命中条件到『description 反模式 + 项目层判定』(B2),让真正的漏采用人群可见。

【本期不处理 agent 侧绑定(N3)】根因里『6 个助理 agent 全 0 skill』属实,但本期不做 agent 侧 skill 绑定/instructions 改写——因为这些 agent 全是 idle 的 local agent,团队当前是直接用各自工具干活、几乎不经 multica agent 派活(全工作区仅 4 个 agent-created issue)。agent 侧绑定留作后续(若团队转向 agent 派活模式再做),本期聚焦人/工具直接可达这条已验证有效的路径。

【排序与验证】先做 1+2,并在荣灿的 Codex/Opencode 环境上验证能真『触发』跑通(他工具最杂、当前最不合规,是最强的可移植性样本);Codex 这条一旦跑通,基本证明全队可移植。再铺 3+4。1 周 appetite,边界:本期只做布线/文档/一个 nudge,不重写状态机、不动 label 设计、不碰 cloud、不做 agent 侧绑定。

【评审迭代】v1 经独立 staff engineer 子 agent 评审,verdict=changes-requested,事实核查全部通过;blocking 三条:B1(架桥验收只测手动调脚本、没测自发触发)、B2(nudge 扩的盲区档只抓标题前缀、漏掉 description 反模式人群,且覆盖率分母未定义)、B3(分层判定规则缺位、未拍板)。v2 已逐条修订(见完成标准 2/5/3 与本 approach),并吸收 N1(2 周观察窗移出 appetite)、N2(并入 CLI 版本核验)、N3(说明 agent 侧绑定为何本期不做)。
