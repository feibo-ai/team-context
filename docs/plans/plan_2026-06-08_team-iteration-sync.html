目标

Goal
建立确定性、安全、分钟级的团队迭代同步机制(本期范围:team-context + team-context-mcp 两仓)。发版(CI绿+tag)后,每台成员机经 launchd 轮询的确定性脚本自动 safe-pull 到最新 release tag;tcmcp-local 从 stdio 迁至 HTTP transport,使 MCP 改动后 rebuild→重启 server 进程→client 自动重连全链路自动完成,彻底消除「改 MCP 必须人工重启 Claude/Codex」的硬边界(Codex 同样受益);每次同步经 notify_team 播报「X 仓→tag vN」。tc-multica CLI 不纳入本期,由现成 multica update + daemon 6h 自检兜底。

完成标准

  • 发版闸门:team-context-mcp 补 tag 驱动 release.yml(已有 main 保护 + 2 tag);team-context 补 release 流程(CI lint+frontmatter 绿为 tag 前置)。两仓各手动打一个 v* tag 能自动产出 GitHub Release。
  • HTTP transport:tcmcp-local 以 HTTP transport 经 launchd 常驻;Claude(--transport http)+ Codex(url=)注册迁完。实测改 packages/local/src/server.ts→脚本 rebuild→client 自动重连、不重启编辑器即见新工具一次。
  • 轮询引擎:DRI 机一条 launchd 脚本跑通完整 Tier1 链路——检测两仓新 release tag → safe-pull → 新增 skill 重铺软链 + 刷新 SKILL.md 物理副本 → registry skill update → multica update → tcmcp-local rebuild+重启 server → notify_team 播报。
  • 安全边界各验一次:脏工作区跳过+告警 / 非目标 tag(或非 main)跳过 / build 失败捕获+通知不强推 / 全程不碰 tc-multica fork 的 upstream。
  • 稳定性:DRI 机连续 3 天无人工干预自动同步,两仓更新分钟级到位、零误覆盖脏区。
  • 收口:ONBOARDING-MEMBER.md:311 裸 git pull 收敛为 ff-only safe-pull;sync-to-multica.sh:33 硬编码 ~/team-context 路径修正。

方案

【总体形态】分层 hybrid:Tier0 软链已免费(git pull 即时生效)→ Tier1 确定性轮询脚本做四类机械活 → Tier2 notify_team 播报。引擎走 launchd 轮询、不挂 multica autopilot——研究已实测确认 autopilot 必走 LLM、无确定性 shell 模式,纯机械同步不该用 LLM 承载。研究量化:团队核心产物约 93% 的 skill/CLAUDE.md 正文是软链进仓库的(Tier0),真正要额外做的只有四类:① 新增 skill 重铺软链 + 刷新会 stale 的 SKILL.md 物理副本 ② registry skill 正文回推 ③ multica update CLI 自更新 ④ tcmcp-local rebuild+重启——全是确定性操作。 【决策 1 · 引擎=launchd 轮询】DRI 确认 self-host(10.0.5.51:8083)无现成 GitHub webhook 入口;且 multica webhook 落点在远程 server、又必经 LLM autopilot,对纯机械同步不合适。launchd 轮询天然 per-machine、无 LLM、与 daemon 30s 轮询同构,代价仅分钟级延迟——对「正文本就软链免费 + MCP 月级才改一次」完全够用。轮询的「源」是各仓最新 release tag(见决策 3),脚本比对本地 vs 最新 tag,有新且 CI 绿则 safe-pull。 【决策 2 · 自动边界=Tier1 全自动(含 MCP),硬边界由 HTTP 迁移消除】原硬约束:stdio MCP 不热重载→改 MCP 必人工重启编辑器(Claude /mcp 可软化、Codex 不行)。DRI 拍板本期就把 tcmcp-local 从 stdio 改 HTTP transport 根治——同仓 remote 已验证该路径(StreamableHTTPServerTransport + express + MCP_TRANSPORT env 切换,health.ts/auth.ts 可复用)。迁移后 server 是 launchd 托管的独立常驻进程,rebuild 只重启该进程、client 自动重连,「人工重启编辑器」硬边界被彻底消除,Tier1 因此可把 MCP rebuild+重启纳入全自动。张力说明:Q2 确认 MCP 仅月级改、auto-rebuild 边际价值有限——但 HTTP transport 的价值不在频率,而在一次性根除唯一人工边界、且对 Codex 同样生效,让整套机制成为干净的确定性闭环,这一战略收益值得本期投入。一次性代价:切 transport 需各成员在本机改一次 client 注册 + 起一个 launchd 托管常驻 HTTP server(端口/健康检查复用 remote health.ts),属 onboarding 级动作、非每次同步。 【决策 3 · 范围=先两仓 + 本期加码两项】tc-multica CLI 自更新已由 multica update(无 sudo、可写目录直接覆盖)+ daemon 6h 自检兜底,边际收益小;且 tc-multica 是 fork、当前卡脏特性分支、有 upstream(multica-ai/multica)——自动同步它风险最高收益最低,明确排除本期(防误碰 upstream)。两仓范围因 DRI 选 tag/release 统一闸门而加码:(a) team-context-mcp 补 tag 驱动 release.yml(已有 main 保护 + 2 tag,补发布即可);(b) team-context 补 release 流程(现 0 tag、CI 仅 lint 文档,以 lint+frontmatter 绿为 tag 前置,tag 标记「已审核快照」)。统一为「只消费 CI绿+tag」后 broken-main 传染风险被 tag 闸门挡住,正是研究风险②的收口。 【安全护栏(写进 safe-pull,研究风险逐条收口)】① ff-only + 仅 main 来源 + 脏工作区跳过+告警(活例:tc-multica 现卡 docs 分支且 ahead 2,对它 pull 必拒,证明设计对);② build 失败捕获+notify、不强推;③ 三套路径不一致(~/feibo/feibo/*、~/team-context-mcp、文档 $HOME/team-context)→ 脚本逐仓显式配路径、严禁通配;④ tc-multica fork 的 upstream 绝不自动碰;⑤ 多机:team 版脚本只在 DRI 一台跑(无本地文件锁,多机同 scope 互相覆盖——研究风险③)。 【复用既有基建(别重造)】team-context/scripts/ 已有 sync-team-config.sh(软链 tc-*+全局 md+registry 推)、sync-to-local.sh、sync-to-multica.sh、team-/my-autopilot.sh + _autopilot-common.sh。本期轮询脚本在其上增量(safe-pull 包裹 + tag 检测 + SKILL.md 刷新 + notify),而非另起炉灶。 【里程碑】M1 两仓发版闸门(关键路径)→ M2 tcmcp-local HTTP transport 迁移+实测自动重连 → M3 launchd 轮询引擎 + Tier1 + Tier2 notify + 安全边界实测 + 3 天稳定性 + 收口。
