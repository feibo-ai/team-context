6 条核查项 · 裁决

核查项裁决实测要点(带证据)
① stdio MCP 是否真不热重载 证实(真硬约束) stdio server 是 client 启动时 spawn 的长驻子进程,整个 session 持有;tsc 覆写 dist 字节对内存中运行的进程零影响。证据:packages/local/src/server.ts:166 new StdioServerTransport() + 文件头 "no remote transport";全仓 0 个 watch/nodemon/chokidar。纠偏 carry-over 说"机制可自动 rebuild"——仓库现状无任何自动 rebuild(dev 也只是裸 tsx 不 watch),自动 rebuild 需另加外部脚本/CI。
② "80% 靠软链免费" 是否成立 方向对 · 口径要修正 纯正文 + 全局 md 的软链覆盖率其实 ~93%(14/15),高于 80%。但"免费"有三个前提被 carry-over 隐去:(1) 仅限已存在条目的正文;(2) 新增 skill 首次出现必须重跑 sync-team-config.sh 才建软链;(3) 每个 skill 的外层 SKILL.md(加载器真正读的触发器/frontmatter)是物理副本、git pull 后仍 stale——实测 tc-1-start 副本 mtime 06-01 18:45 落后源最后 commit 06-02 10:38(差 3 行,缺 rule #6)。纠偏 30 个 ljg/gsap 软链指向个人仓 ~/.agents(非团队三仓),不计入本题分母。
③ multica autopilot/trigger/daemon 能力边界 已摸清 webhook 在远程 server 接收(/api/webhooks/autopilots/{token}),非本地 daemon——self-host 10.0.5.51:8083 内网可达即可;cron 标准表达式 + IANA 时区,分钟级,server 端调度、不依赖本地 daemon 在线;daemon per-machine,30s 轮询 + WebSocket 唤醒(WS 断退回轮询)。关键 autopilot 必然产出一个 LLM agent task,无"跑确定性 shell"模式(两种 mode 终点都把 prompt 交给 Claude/Codex)→ 纯机械同步活不该用 autopilot 承载。
④ tcmcp-local build/产物/启动 已确认 pnpm monorepo,根 build = pnpm -r --filter './packages/*' build,本地 server 是子包 @tcmcp/local,构建器 tsc。纠偏 产物是 packages/local/dist/server.js,不是 carry-over 写的 dist/server.js(~/.codex/config.toml 还留有旧死路径已删的注释,印证踩过此坑)。Claude Code 与 Codex 都以 stdio node .../packages/local/dist/server.js 注册。
⑤ 三仓 CI 绿/tag/release 现状 分仓裁决 见下方"三仓现状表"。一句话:只有 tc-multica 今天就能"CI绿+tag"(tag 驱动 release、几乎日更);team-context-mcp 半能(main 分支保护=绿,但 0 release、无发布节奏);team-context 不能(0 tag、CI 仅 lint 文档合规、不 build/test)。
⑥ 存量脚本已做到哪步 已复核 sync-*.sh 全部手动触发(无 cron/hook);team-/my-autopilot.sh + _autopilot-common.sh 全程零 git 操作(只建 autopilot,内置 PB-04 guardrails lint);git pull --ff-only 只出现在 onboarding 文档(且已是 --ff-only 2>/dev/null || true fail-open)。当前没有任何可执行的自动 pull 在跑——同步风险是"前瞻约束"不是"现存 bug"。

B · stdio MCP 热重载硬约束 与 破解路径

Findings — MCP reload
  • 裁决:硬约束为真。机制见核查项①。本地改 MCP 代码 → 必须 tsc rebuild + 重启 client 才生效;机制能自动 rebuild,但无法替你重启正在跑的 Claude/Codex。
  • 破解 1(最低成本,部分软化):Claude Code 交互内 /mcp 斜杠命令可重连单个 server(触发重新 spawn 子进程→载入新 dist),不必重启整个编辑器。但仍需一次手动动作,且 claude mcp 子命令本身无 restart/reconnect。
  • 破解 2(根治,代价中):把 tcmcp-local 从 stdio 改 HTTP/SSE transport——同仓 remote 已证明该路径(StreamableHTTPServerTransport + express,MCP_TRANSPORT env 切换;Claude --transport http、Codex url= 都支持)。改后 server 独立进程,rebuild 只重启 server、client 自动重连。代价:本地起常驻 HTTP 进程 + 端口/健康检查(remote 已有 health.ts/auth.ts 可复用)。
  • 破解 3:tcmcp-remote(Zeabur)本质已规避——server 端部署、push main 重建、client 走 HTTP 自动重连,不经本地 spawn。但 remote 与 local 工具集不同(remote 偏飞书/协作,local 偏 plan/case/git 本地操作),非简单换 transport 即可合并。
  • 差异 Codex 受约束更硬:无 /mcp 式单 server 重连,只能重启 Codex 或靠 startup_timeout_sec。

C · multica CLI 现成零件的能力与边界

Findings — multica CLI (v0.4.10, 源码核实)
  • update:纯 GitHub Release 自更新(写死 feibo-ai/tc-multica/releases/latest),semver 比较,下载→校验→os.Rename 原地替换。无 sudo、无 brew 分支;装在可写目录(/opt/homebrew/bin)直接覆盖,不可写则 rename 报错(不提权)。daemon 另有每 6h 自动检查 CLI 新版。
  • autopilot:调度单元,必走 LLM agent。create 必填 --agent+--mode(create_issue/run_only),--description 是给 agent 的自然语言 prompt,非命令。trigger-add --kind schedule|webhook:schedule 需 --cron+--timezone;webhook 自动生成 awt_ bearer token URL。
  • skill:操作 server 端 registry,非本地文件同步。import --url 单向(外部→registry);update <id> 改 registry DB 记录。无"本地目录↔registry 双向同步"原语。
  • repo checkout <url>:从 daemon 的 bare-clone 缓存创建 git worktree(--ref 指定 ref)。是 checkout/worktree 语义,不是在已有工作树里 fast-forward,不能当通用 git pull 用。
"不做" 清单(这些零件覆盖不到)
  • 任意仓的 git pull --ff-only / 已有工作树 fast-forward
  • 任何 build/编译(pnpm build/tsc/go build)
  • 重启编辑器 / 重启 daemon 之外的进程
  • 以确定性 shell 脚本作为调度产物(autopilot 必走 LLM,无 "run a command" 型 trigger)
  • webhook 本地接收(必经 server;daemon 本地只有 127.0.0.1 health listener)

Carry-over 复核结论

Confirmed / Corrected / New
  • 证实 stdio MCP 不热重载是真硬约束;local MCP 这格只能"自动 rebuild + 通知重启"。
  • 证实 multica CLI 是"零件库",不解决 git pull / build / 编辑器重启;daemon per-machine。
  • 证实 自动 pull 对脏区/非 main 的风险(ff-only+脏跳过的设计方向对)。
  • 纠偏 "80% 免费":正文软链其实 ~93%(更高),但 SKILL.md 触发器是会 stale 的物理副本、新增 skill 不免费、registry 需 CLI——"免费"被偷换了概念。
  • 纠偏 "机制可自动 rebuild":仓库现状无任何自动 rebuild,需另建。
  • 纠偏 产物路径是 packages/local/dist/server.js(monorepo),非 dist/server.js。
  • 纠偏 "只消费 CI绿+tag" 不能一刀切:仅 tc-multica 今天可落地,另两仓缺发版节奏。
  • 新增 autopilot 必走 LLM(无确定性 shell 模式)→ 纯机械同步别塞 autopilot,用确定性脚本。
  • 新增 webhook 落点在远程 server(self-host 内网可达即可);cron 由 server 端调度不依赖本地 daemon 在线。
  • 新增 三套路径不一致 / tc-multica 是 fork / multica-cli skill 是孤儿副本 / sync-to-multica.sh 路径硬编码——四个落地暗坑。

可选方案(为 Plan 三决策准备 · 非定论)

以下是带证据的选项与权衡,供 Plan session + DRI 拍板。本 Research 不做决策。
决策 1 · 引擎:GitHub webhook vs launchd 轮询
  • 选项 A · webhook 触发:即时。但 multica 的 webhook 落点是远程 server 且 autopilot 必走 LLM——若要纯机械执行,webhook 更适合打到"本机轻量 listener / CI 触发脚本",而非 multica autopilot。每仓加一步配置,需 server 可达入口。
  • 选项 B · launchd 轮询:本机定时跑确定性脚本,简单、无 LLM、天然 per-machine,与 daemon 30s 轮询同构;代价是分钟级延迟。
证据倾向(非定论):纯机械同步(safe-pull/rebuild/CLI update)用确定性脚本更稳;webhook 即时性对"skill 正文"收益有限(本就 git pull 免费),主要价值在"MCP 变更需重启"的即时通知。引擎可能是"轮询执行 + 通知用 webhook/notify_team"的组合。
决策 2 · 自动边界:Tier1 全自动 vs 通知+一键确认
  • 可确定性自动(低风险):safe-pull(ff-only + 仅 main + 脏则跳过告警)、新增 skill 重铺软链 + 刷新 SKILL.md 副本、registry skill update、multica update(无 sudo、可写目录直接覆盖)。
  • 硬边界(不可全自动):本地 MCP 变更→必须人工重启编辑器(Claude 可 /mcp 重连软化,Codex 不行)→ 这格只能"自动 rebuild + 通知重启"。
证据倾向(非定论):Tier1(safe-pull+rebuild+CLI update)风险可控可全自动;但 broken main 对 team-context(无闸门)仍是隐患 → 自动边界与"决策 3 范围/闸门"耦合,team-context 在补上发版/CI 闸门前,自动 pull 宜更保守。
决策 3 · 范围:先两仓 vs 含 tc-multica CLI
  • 选项 A · 先两仓:team-context 同步几乎纯软链(正文免费),增量只是"新增 skill 重铺 + SKILL.md 刷新";team-context-mcp 是真正难点(rebuild+重启)。两仓覆盖日常绝大多数痛点。
  • 选项 B · 含 tc-multica CLI:CLI 自更新已由 multica update + daemon 每 6h 自动检查解决,边际收益小;且 tc-multica 是 fork、在脏特性分支、有 upstream——自动同步它风险最高、收益最低。
证据倾向(非定论):先两仓;tc-multica CLI 用现成 multica update 兜底即可,不必纳入自建同步。
