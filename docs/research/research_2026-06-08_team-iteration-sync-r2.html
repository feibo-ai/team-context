问题

Question
第一轮 Research+Plan 后,Plan 阶段 DRI 拍板 pivot:本期不要本地 MCP。核心新问题——能否把 @tcmcp/local 的 13 个工具迁到 skills + multica CLI + agent 自己跑(git / 写文件 / 建 issue),本期消除本地 MCP server?是否可行、成本几何、什么会破?新方向不是「怎么同步本地 MCP」,而是「怎么消除它让同步变简单」。本 session 只测绘、不决策(决策属下一轮 Plan)。
方法:主 session 亲读渲染管线源码(render/*.ts + doc_publish.ts + 代表性 create 工具)建立第一手判断;并行派 4 个只读子代理分头实测——① 13 工具分类 ② CLI 覆盖度 + MCP 注册 ③ 校验链路 + SOP 纪律 + token 成本 ④ 残余同步面 + 引擎重估 + 迁移成本。所有结论带 file:line 证据;与第一轮一致标 证实,不符标 纠偏,新增标 新增。
一句话裁决(供 Plan)
去 local MCP 技术可行,且是同步问题的「釜底抽薪」——本机侧 MCP 同步难点归零,launchd 引擎与 tag 闸门双双降级。唯一硬约束:doc_publish 的「上传 HTML + !file 内联渲染」现成 CLI 单命令做不到,必须靠 agent 跑一步 raw HTTP(或给 CLI 加 flag)——这是可行性的关键前提,不是 blocker。纪律不退化、渲染可保真,代价是 5 个 skill + 8 个 autopilot 改写(~6-9 人日)。

B · CLI 覆盖度 与 pivot 命门(子问题③)

Findings — CLI coverage(v0.4.10)
MCP 写操作对应 CLI覆盖度缺口 / 证据
createIssueissue create全字段齐(--title/--description/--project/--attachment…)
updateIssueissue update / issue status全但无 --attachment-ids,不能像 MCP 那样 PUT 时绑附件
commentOnIssueissue comment add纯文本全--content/--content-file/--content-stdin 齐;但带渲染附件做不到(见下)
addLabel/removeLabelissue label add/remove部分纠偏 CLI 收 <label-id> UUID 不收 name;MCP 的 name→UUID 解析(getLabelId)转嫁 agent(须先 multica label list)
uploadFile(无独立命令)部分multica attachment 顶层只有 download;upload 仅作 create/comment 的 --attachment 附属,且吞 url
publishDoc无单命令等价缺口(命门)见下方核心问题
核心问题:doc_publish 能否用 CLI 复现内联渲染?——不能
  • publishDoc = 三件事原子完成(multica-client.ts:152-167):① 上传 HTML 拿 {id,url} → ② 评论体拼 !file[name](url)(依赖 url 做内联 token)→ ③ 建评论并 attachment_ids 绑定。
  • CLI issue comment add --attachment 实测(cmd_issue.go:1070-1142):content 原样透传、无一行拼 !file;调的是 UploadFile(client.go:383 decode 只取 result["id"]),不是 UploadFileWithURL。
  • 后果:没有 !file token 的附件「has NO render surface, shows nowhere」——HTML 不会内联渲染。upload-file 服务端响应确实带 url(handler/file.go:66 URL: a.Url),但在 CLI 命令层被吞,agent 无从获取。
  • 纠偏 doc_publish.ts:28-30 注释「CLI 的 attachment 只能 download 不能 re-upload」已过时:能 upload,但不能内联渲染才是真约束。
  • 复现路径二选一(均超出单条 CLI):(1) agent 保留一步 raw HTTP——直接 POST /api/upload-file(响应带 url)→ 自拼 !file → comment 带 attachment_ids;(2) 改 CLI——给 comment add 增「上传后自动拼 !file」或让 UploadFile 回 url。

Findings — 渲染 token 成本 与 skill_lint 红线

F · 迁移成本 与 什么会破(子问题⑦)

深化追问 · 机理与可行性(DRI 第二轮追问)

① 为什么不能内联渲染

内联渲染的唯一触发器 = 评论正文里的 !file[文件名](url) file-card token(multica 前端 preprocessFileCards → file-card → HtmlAttachmentPreview 才渲染,multica-client.ts:4-12)。光上传 + 绑 attachment_ids 没 token 就「无渲染面」;拼 token 需要 url。
  • MCP 工具能内联:TS 端 uploadFile 取 url(multica-client.ts:196)→ publishDoc 拼 fileEmbed(:162-163),无 url 则降级提示(:164)。
  • CLI issue comment add 不能:用旧 UploadFile(client.go:338)只返回 id、吞掉 url(:383),正文原样透传不拼 !file(cmd_issue.go:1123)。
  • 根因 不是后端缺能力(/api/upload-file 响应本就带 url),是 CLI 这条命令没接「取 url + 拼 token」两步。

② CLI 二次开发可行性 —— 极小,同仓已有范例

命门最小修复
  • UploadFileWithURL(client.go:393,返回 (id,url,err) 不吞 url)已存在,且 CLI 自己已在用(cmd_agent.go:773 头像上传)。
  • 修复 runIssueCommentAdd(cmd_issue.go:1115):改用 UploadFileWithURL 拿 id+url → 正文拼 !file[名](url)(格式 multica-client.ts:14-16)→ comment 带 attachment_ids 绑定 → 加 --inline flag。只动一处 ~5-10 行。
  • 注:UploadFileWithURL 不绑 issue_id,靠 comment 的 attachment_ids 绑定即可。

④ CLI 自动 update —— 一个 env 开关

daemon 有自更新能力,self-host 默认关(config.go:381 isOfficialCloudServer→false)。设 MULTICA_DAEMON_AUTO_UPDATE=true(config.go:382)即开,每 6h 在 idle 时自更新(auto_update.go:43)。进一步削弱「需专门同步引擎」的必要性。
