MODULE 01 · PRINCIPLES

</div>

7 条不可妥协的\
核心原则
===============

所有动作的上位依据 · 当 SOP 没覆盖某场景时,用这 7 条裁判 ·
它们是**不可妥协的** —— 当具体动作和某条原则冲突时,以原则为准。

<div class="module-meta">

READ TIME · **~ 8 分钟** AUDIENCE · **全员必读** UPDATE · **罕见**

</div>

</div>

<div class="principle">

<div class="principle-num">

01

</div>

<div class="principle-content">

### Research / Plan / Implement 严格分离

不要让 Claude 在研究的同时规划 · 在规划的同时实施。三个 discrete
sessions · 三个不同的 prompt · 中介是 **Markdown plan 文件**。

具体动作见
<a href="#p-03" style="color: var(--accent);">03 · RPI 框架</a>。

<div class="evidence">

参考:Sam Brickman / Ambral CTO · Dexter Horthy · HumanLayer RPI 框架

</div>

</div>

</div>

<div class="principle">

<div class="principle-num">

02

</div>

<div class="principle-content">

### Context 浑浊时立刻重启 · 不要 fix

Claude 的 context window 用太满会进入 **"dumb zone"** · 输出质量下降 ·
但自己感觉不到。<u>识别信号 + 立刻重启</u>是工作纪律。

**常见 context 浑浊信号**:

- Claude 开始说 `"You're absolutely right"` · 过度迎合
- 重复之前已经否定的方案
- 答非所问 · 或者抓不住重点
- 解决"上一个问题的衍生" · 而不是当前问题

看到任一信号 → **新开 session** · 把关键信息重新交付。具体的 context
阈值由个人判断(不同任务、不同工具默认不同)。

<div class="evidence">

参考:Dex Horthy 100k+ session 分析 · AI Engineer Code Summit 2025

</div>

</div>

</div>

<div class="principle">

<div class="principle-num">

03

</div>

<div class="principle-content">

### 并行 Agent 是工作节奏 · 大致 5-10 active

不是"偶尔多开几个" · 是**同时 active 多个 Claude**(团队大致节奏)·
每个人可以配 worktree 长期挂着 · 按需启动。

5 人团队 = 25-50 个 Agent 单元并行运行 · 这是 AI Native
工作方式的基本物理量。

具体每个人开多少 · 按自己舒服的节奏。**避免极端**:1-2 个太少(没用上 AI
Native)· 50+ 太多(质量崩溃)。

<div class="evidence">

参考:Boris Cherny 个人日常 · HumanLayer CodeLayer · GitHub Issue \#42796

</div>

</div>

</div>

<div class="principle">

<div class="principle-num">

04

</div>

<div class="principle-content">

### "Start over" 优于 "fix"

当 Claude 走偏 → 中断 + 重启 · **不要试图纠正**。10-20% 的 session
直接放弃是健康的。

"卡 30 分钟还没进展 → start over" 是个有用的 reminder ·
具体阈值按个人判断。

<div class="evidence">

参考:Boris Cherny · Anthropic Data Science "slot machine" 模式

</div>

</div>

</div>

<div class="principle">

<div class="principle-num">

05

</div>

<div class="principle-content">

### CLAUDE.md 是高门槛抽象层 · cases/ 是无限累积具体层

沉淀分两个根本不同的世界:**抽象**(给 Agent · 必须精炼)vs **具体**(给人 +
Agent 按需 · 可无限累积)。

CLAUDE.md
进入门槛:**这条规则适用于<u>未来所有</u>类似项目**(不是只这个项目的具体情况)·
建议控制在 2-3k token。

具体项目 debrief 写到 `cases/YYYY-MM-DD-name.md` · 无限累积 OK。

详见 <a href="#p-04" style="color: var(--accent);">04 · 沉淀三层</a>。

<div class="evidence">

参考:Boris Cherny CLAUDE.md = 2.5k token · 每周更新多次 · Anthropic
内部团队 + Sanity 6 周实战

</div>

</div>

</div>

<div class="principle">

<div class="principle-num">

06

</div>

<div class="principle-content">

### Plan Mode 不可妥协 · 不允许 vibe code

每个非平凡任务**必须先生成 plan 文件**(Markdown · 可读 · 可审 · 可改)·
通过第二个 session 做 review · 只有 plan 通过才进入执行。

任务层至少 3 句话 plan · 项目层完整 plan markdown · **永远不允许 "vibe
code"**。

这是 **2 个不可妥协之一**。

<div class="evidence">

参考:Boris Cherny · Dex Horthy "Do Not Outsource The Thinking"

</div>

</div>

</div>

<div class="principle-num">

07

</div>

<div class="principle-content">

### 项目/任务结束必有 Debrief

提炼出的<u>通用规则</u>才进 CLAUDE.md · 具体记录留 case。

"做完就走 · 不写 debrief" = 项目没结束。debrief 不是流水账复盘 ·
是<u>拆解关键判断的因果链</u>。

具体动作见
<a href="#p-03" style="color: var(--accent);">03 · RPI 框架</a>
收尾阶段。

这是 **2 个不可妥协之二**。

<div class="evidence">

参考:Sanity 工程团队 / Anthropic 内部 review 实践

<div class="sub-section">

## 两层粒度 · 项目 vs 任务

所有新事情先判断属于哪一层 ——
**唯一的判断标准是:它是不是一个独立的大目标?**

| 维度 | Project · 项目层 | Task · 任务层 |
|----|----|----|
| 定义 | 一个独立的大方向、大目标 · 本身有完整存在意义 | 项目下的具体事项 · 必须挂在父项目下 |
| 例子 | AI 招聘平台 / 要抱抱 / Team Context 自身迭代 | debug 某个 bug / 优化某个 prompt / 处理用户反馈 |
| RPI 走法 | 完整走 Research → Plan → Implement | **可跳过 Research** · 直接 Plan + Implement |
| plan 长度 | 完整 plan markdown(几百字) | 至少 3 句话 |
| DRI | 必须有 DRI · 拥有项目全权 | 通常自己执行 · 默认就是 DRI |
| debrief | 写完整 case file(5 个 section) | 写简短 case file 或合并到当周 |

</div>

<div class="sub-section">

## Plan Markdown 必备字段

无论项目还是任务 · plan markdown 必须包含以下 **4 个字段**(替代 v0.1
的"6 问"):

| 字段 | 含义 | 不能是 |
|----|----|----|
| 目标 | 这件事做完要达到的具体结果 · 可被描述、可被检验 | "做得更好" / "搞定它" |
| 完成标准 | 可观测信号 · debrief 时的对照基准 · 可被 Outcomes 自检 | "做好为止" / "没问题就行" |
| 怎么分 | 项目层:DRI + 谁参与;任务层:就是自己 | "看情况" / "大家一起搞" |
| 时间盒 | 项目层:appetite(几天/一周/一月);任务层:估计小时数 | "尽快" / "越快越好" |

</div>

<div class="sub-section">

## 三阶段总图

所有事情走这三个阶段。**三阶段是骨架,每个阶段都是 discrete
sessions,不是一个长对话。**

| 阶段 | 名称 | 核心动作 |
|----|----|----|
| PHASE 01 | <span class="phase-tag r" style="margin:0;">RESEARCH</span> + PLAN | 声明意图 → Research session → Plan session → 第二个 session review → DRI 拍板 → 广播 |
| PHASE 02 | <span class="phase-tag i" style="margin:0;">IMPLEMENT</span> | 并行 Opus 4.7 sessions + 监督 CoT + Task Budget + 卡 30 分钟 start over + 微更新 |
| PHASE 03 | <span class="phase-tag p" style="margin:0;">DEBRIEF</span> | Agent 写 debrief 草稿 → 写入 cases/ → DRI review 校准 → 判断通用规则提升到 CLAUDE.md |

有清晰输入/输出/验证标准 · 输出到

<div class="step">

<div class="step-num">

STEP 4 · Review

</div>

<div class="step-title">

第二个 session · Plan 审查

</div>

在第二个 Claude Code session(Opus 4.7)运行:

<div class="code-block">

"我刚做完一个 plan · 文件在 docs/plans/plan\_\[date\]\_\[topic\].md\
请你以 staff engineer 角色 review · 重点指出技术风险 / 不一致 /
被忽略的边界情况 / 过度工程"

<div class="step-num">

<div class="step-title">

输出:plan markdown 修订到 v0.x · 标记"已批准 · 可执行" ·
这是**整个启动阶段唯一需要人专注思考的环节**。

<div class="step">

<div class="step-num">

DAILY 01 · 早上

<div class="step-title">

启动 5-10 个 Claude

- 给每个 Claude 派一个 phase · 启动 Claude Code session(全程不区分阶段)

<div class="step">

<div class="step-num">

DAILY 02 · 监督

</div>

<div class="step-title">

监督 chain of thought · 手指放 ESC 上

</div>

**关键时刻:当 Claude 开始一个任务的前 30 秒**

- 盯它的 chain of thought · 看方向对不对
- 方向不对 → **立刻 ESC 中断** · 不要让它跑完错误方向
- 看到 "You're absolutely right" → 立刻 /clear 或新启 session

<div class="evidence">

Vulcan CEO Tanner Jones · YC S25 · "前几次 tool call 抓到走偏比让 Claude

</div>

<div class="step">

<div class="step-num">

DAILY 03 · 卡住

</div>

<div class="step-title">

遇到错误 · start over · 不要 fix

</div>

**当 Claude 第一次没做对时**:

- 不要"和它讨论怎么改对"
- /clear 或开新 session
- 把 plan markdown 改得更清楚
- 让新 session 重新执行

**10-20% 的 sessions 被放弃是健康的** · 这不是失败 · 是 AI Native
工作方式。

</div>

<div class="step">

<div class="step-num">

DAILY 04 · 微更新

</div>

<div class="step-title">

中午 5 分钟 · CLAUDE.md 微更新

</div>

把上午发现的 Claude 反复犯错加进 CLAUDE.md · 立刻 git commit + push ·
团队同步

格式:*"❌ 不要 X · 因为 Y · 应该 Z"*

注意:微更新前先判断 ——
这条是<u>通用规则</u>还是<u>这个项目独有的</u>?后者只进 case file · 不进
CLAUDE.md。详见
<a href="#p-04" style="color: var(--accent);">04 · 沉淀三层</a>。

</div>

<div class="step">

<div class="step-num">

DAILY 05 · 收工

</div>

<div class="step-title">

傍晚 · Agent 写 daily summary

</div>

Agent 基于今天的决策日志 + commits 自动生成 daily summary · 写入 Team
Context · 飞书群推送 · 人 30 秒过目。

包含:① 完成的 phases · ② 启动的 sessions · ③ 放弃的 · ④ 决策日志 · ⑤
卡在哪里

原则:进度衡量来自**工作产物本身**(commit / decision log / 完成的 phase)
· 不是"主动汇报"。

## PHASE 03 · 收尾 · debrief 流程

<div class="quote">

debrief 不是流水账复盘 · 不是"列出做了什么" ·\
是**拆解关键判断的因果链** · 提炼出下次能复用的东西。

</div>

每个项目/任务结束都要写 case file · 但**只有 10% 的发现会进 CLAUDE.md**
· 90% 留在 case 文件作为具体记录。

<div class="step">

<div class="step-num">

STEP 1

</div>

<div class="step-title">

Agent 自动生成 debrief 草稿

</div>

项目/任务完成时 · 启动一个新 Claude Code session:

<div class="code-block">

"请阅读 docs/research/\[date\]\_\[topic\].md ·
docs/plans/\[date\]\_\[topic\].md · 这个项目相关 commits · 生成 debrief
草稿写到 cases/YYYY-MM-DD-\[project-name\].md。\
\
必须包含 5 个 section:\
① 目标(plan 里原始目标)\
② 实际做的事(压缩时间线 · 不超过 200 字)\
③ 完成标准 + 是否达成\
④ 关键判断分析 — 真实成败原因 · 不要场面话\
⑤ 通用规则候选 — 适用于未来所有类似项目?如有列出 + 标记'需要 DRI
决定是否提升'"

</div>

</div>

<div class="step">

<div class="step-num">

STEP 2

</div>

<div class="step-title">

DRI review · 校准真实成败原因

</div>

DRI 读 Agent 生成的 case file 草稿 · 重点 review section ④:

- Agent 抓到的"成败原因"是真实的还是表面的?
- 有没有 Agent 没看到的关键判断点?
- "如果重来一次 · 哪个判断会改?"
- "古法不可能"在哪一步发生了?

这是**整个 debrief 唯一需要人专注思考的环节** · Agent 草稿不替代这一步。

</div>

<div class="step">

<div class="step-num">

STEP 3

</div>

<div class="step-title">

DRI 决定:提升到 CLAUDE.md?

</div>

对 section ⑤ 的每个"通用规则候选" · DRI 用 1 个问题判断:

<div class="callout">

**这条规则适用于<u>未来所有</u>类似项目吗?**\
(不是只这个项目的具体情况)\
\
✅ YES → 提升到 CLAUDE.md · 同时保留 case file ·
写成抽象规则(不带具体项目名) · 控制总 token \< 3k\
❌ NO → 只留 case file · 不进 CLAUDE.md · 累积无上限 OK

</div>

</div>

<div class="step">

<div class="step-num">

STEP 4

</div>

<div class="step-title">

项目层:周五 demo

</div>

仅项目层需要 · 周五下午 30 分钟 · DRI demo 这个项目的结果 · 全员看 ·
不需要 PPT · 直接演示真实产物。

原则:demo 是**对结果的庆祝** · 不是"汇报会" · task 级别不需要 demo。

沉淀三层\
L1 抽象 · L2 具体 · L3 中间
===========================

沉淀分两个根本不同的世界:**抽象**(给 Agent · 必须精炼)vs **具体**(给人 +
Agent 按需 · 可无限累积)· v0.2 之前混在一起 · v0.3 严格分开。

<div class="sub-section">

## 三层架构 · 一张总图

| 层 | 资源 | 大小 | 进入门槛 | 用途 |
|----|----|----|----|----|
| L1 · 抽象 | **CLAUDE.md** | **硬上限 3k token** | **极高** · 适用于未来所有类似项目 | 每个 Claude session 自动加载 |
| L2 · 具体 | **cases/** | 每个 1-3k · 无上限累积 | **低** · 每个项目结束都写 | 人月度 review + Agent 按需主动查 |
| L3 · 中间 | .claude/skills/ | 每个 ~500 token | 被验证过的具体 prompt | Claude 按需调用 |
| L3 · 中间 | standards/ | 累积 | 跨项目的判断标准 | 人检索 + Agent 偶尔读 |
| L3 · 中间 | decisions/ | 累积 | 非平凡架构决策 | 人检索追溯历史 |
| L4 · 档案 | docs/research/ · docs/plans/ | 每个 1-5k | 项目启动时 | 项目执行依据 · 用过即归档 |

<div class="callout">

**核心区别**:L1 是高门槛 / 硬上限 / 给 Agent · L2 是低门槛 / 无上限 /
给人 · 这两层是 v0.3 沉淀架构的<u>根本设计</u> · 其他层是支撑。

</div>

## 提升路径 · case → CLAUDE.md

每个项目 debrief 时 · DRI 判断 case 里的"通用规则候选"能否提升:

| 步骤 | 动作 | 输出 |
|----|----|----|
| case 写完 | section 5 列出"通用规则候选" | 候选清单(0-3 条) |
| DRI 判断 | 逐条问:"适用于未来所有类似项目吗?" | YES / NO 标记 |
| YES | 提炼成抽象规则(去掉项目名)· 加进 CLAUDE.md | CLAUDE.md +1 条 · case 也保留 |
| NO | 不动作 · 具体记录已经在 case 里 | case 保留 · CLAUDE.md 不变 |

<div class="callout green">

**经验数据**:每个项目大约 **10% 的发现**会被提升到 CLAUDE.md · 90% 留在
case 文件作为具体记录 · 这是健康比例。

## DRI · Directly Responsible Individual

<div class="quote">

每个项目有一个 DRI ·\
DRI 拥有这个项目的**整个**可见性、决策权、最终结果责任 ·\
不像很多公司分"产品 DRI + 工程 DRI + 设计 DRI" —— 我们一个项目就一个
DRI。

<div class="quote-attribution">

— Sulman Choudhry · OpenAI ChatGPT 工程团队 · 2026-03

</div>

</div>

### DRI 的三个属性

| 属性 | 含义 |
|----|----|
| 可见性 | 对项目所有动作有**完整可见性** · 所有 plan / decision / commit / case 都能看到 |
| 决策权 | 所有判断的**最终决策权** · 包括 review 别人的工作 |
| 责任 | 从启动到收尾的**每一步**都对项目负责 · 包括失败的后果 |

</div>

<div class="sub-section">

## 职责的四种类型

每个项目启动时 · 把工作按下面四种职责类型拆解 ·
然后由团队成员认领。**同一人可承担多种职责 ·
同一职责也可由多人共同承担**。

| 类型 | 是什么 | 对什么负责 | 什么时候出现 |
|----|----|----|----|
| <span class="role-pill dri">DRI</span> | 项目全权负责 · 可见性 + 决策权 + 责任 | 项目从启动到收尾的整体 | **每个项目有且仅有一个** |

| 傍晚 5 分 | Agent 自动写 daily summary · 推送飞书群 |

1.  停下当前执行
2.  启动新 Claude session 重新 Plan(可能跳回 Research)
3.  升级 plan markdown 到 v1.x
4.  让第二个 session 重新审查
5.  DRI 拍板 · 继续执行

原则:plan 是<u>活的</u> · 但每次修订都走 review · 不要 "vibe code"。

</div>

修 bug\
· 参考示例
==========

场景:debug 某个具体 bug / 小优化 / 处理用户反馈 / 单文件修复 · 可以跳过
Research · 直接 Plan + Implement · 大致 1-3 小时闭环。
下面是一种参考方式 · **不是必须按这个走**。

<div class="module-meta">

SCENARIO · **任务层 · 单一明确问题** DURATION · **大致 15 分钟 - 3
小时**

</div>

<div class="sub-section">

## 什么时候用这个 playbook

| ✅ 用 bug-fix playbook   | ❌ 不要用 · 该走别的 playbook       |
|--------------------------|-------------------------------------|
| 某个页面按钮失效         | 整个页面重设计 → greenfield         |
| 一段代码报错             | 整个模块架构问题 → greenfield       |
| 用户反馈某个文案错       | 整体改产品 voice → greenfield       |
| 调一个 prompt 的输出格式 | 重做 prompt 体系 → greenfield       |
| 批量改 10 个相似文件     | 跨模块依赖的 feature → Level 3 Team |

<div class="callout">

**核心判断**:范围小(1-3 文件) + 问题明确(知道改什么) + 无跨模块依赖 →
bug-fix playbook

</div>

## 4 步流程 · 总耗时 15 分钟 - 3 小时

<div class="step-card">

<div class="num">

STEP 1 · 2 分钟

</div>

<div class="title">

写 mini-plan(3 句话)

</div>

飞书发自己留底 + 写到 plan 文件第一行:

<div class="code-block">

任务:\[一句话说清楚\]\
完成标准:\[1 个可验证信号\]\
边界:\[不要碰什么\]

</div>

不是项目层 · 不需要广播 · 不需要 DRI 拍板 · 自己就是 DRI。但 mini-plan
必须有 · **不允许 vibe code**(非妥协 1)。

<div class="step-card">

<div class="num">

STEP 2 · 主体时间

</div>

<div class="title">

单 session 执行 · Opus 4.7

</div>

启动 1 个普通 Claude Code session(<u>不需要</u> Agent View · 不需要
Level 3):

<div class="code-block">

\[prompt 三条铁律 + 通用模板\]\
\
任务:\[mini-plan 第一行\]\
具体动作:\[动词+数字+具体值\]\
输出:\[diff / 修改的文件 / 测试通过\]\
边界:\[只改 X · 不要碰 Y · 发现别的问题写 TODO\]

</div>

不需要 subagent · 不需要并行。

</div>

<div class="step-card">

<div class="num">

STEP 3 · 5 分钟

</div>

<div class="title">

人 review + 合并

</div>

读 diff · 跑测试 · 验证完成标准 · git commit · push。

不需要项目层那种 review · 但**人必须亲眼看 diff** · Claude ship
的代码就是你 ship 的(原则 Sanity)。

</div>

<div class="step-card">

<div class="num">

STEP 4 · 2 分钟

</div>

<div class="title">

轻量 debrief(必须做)

</div>

非妥协 2 · 任务也要 debrief · 但**极简版**:

<div class="code-block">

追加到 cases/YYYY-MM-WW-tasks.md(按周合并所有小任务):\
\
\## \[今天日期\] · \[任务标题\]\
- 做了什么:\[1 句话\]\
- 完成标准是否达成:\[YES / 部分 / NO\]\
- 通用规则候选:\[有 / 无 · 有的话列出\]

</div>

单独任务 \< 1 小时的 · 不开新 case file · 合并到当周
`cases/YYYY-MM-WW-tasks.md` · 月底 review。

</div>

</div>

<div class="sub-section">

## 卡住了怎么办

| 情况 | 动作 |
|----|----|
| Claude 走偏 | 立刻 ESC · 不要看它跑完错误方向 |
| 卡 30 分钟没进展 | /clear 重启 · 升级 mini-plan(可能要加更多边界 / 更精确动作) |
| 第二次还是错 | **升级到 greenfield playbook** · 说明问题比想象复杂 · 需要 Research |
| 看到 "You're absolutely right" | 立刻 /clear · 这是 context 浑浊信号 |

<div class="callout accent">

**升级信号**:如果一个 bug-fix 任务跑超过 3 小时还没解决 ·
说明它<u>不是任务层</u> · 应该升级成项目走 greenfield playbook。\
**不要硬撑** · 误判任务粒度是常见错。

</div>

</div>

<div class="sub-section">

## 容易踩的坑

- **不写 mini-plan 直接喂 Claude** → 反 pattern · 非妥协 1 违反 · "vibe
  code"
- **不亲眼看 diff 直接 merge** → 你 ship 的代码 · 错就是你的
- **跳过 debrief** → 一个月跑了 100 个小任务 · 一个学习都没沉淀
- **用 Agent View / Level 3 跑小 bug** → orchestration 错配 · 浪费时间 +

多 Agent 协同\
· 参考示例
==============

当一个 feature 涉及多个独立模块、并且彼此有交付依赖时 · 可以考虑让多个
Claude session 并行 + 一个 lead session 协调。 下面是一个真实示例 ·
**不是必须按这个走** · 是给团队第一次想试时的起点参考。

<div class="module-meta">

SCENARIO · **多模块协同 feature** STATUS · **Aaron piloted ·
团队还没正式用过**

</div>

</div>

<div class="sub-section">

## 什么时候<u>可以考虑</u>多 agent 协同

这不是规定 · 是<u>识别信号</u>:当下面几条都成立时 · 多 agent
协同<u>可能</u>比单 agent 顺序做更快。

- 一个 feature 涉及 **3-5 个相对独立的模块**(例:backend + frontend +
  tests + review)
- 模块之间有清晰的交付依赖(谁完成什么 · 谁等谁)
- 你能<u>说清楚</u>每个 agent 各自做什么(说不清楚就不是这个场景)
- 预计 1-4 小时能闭环 · 不是几天的大项目

<div class="callout">

**反信号**(说明该用别的方式):3 个独立 feature 没依赖 → 各自单 session
就行 / 单文件 bug → 直接修 / 不确定方向 → 走 greenfield。

</div>

</div>

<div class="title">

</div>

- 跑测试 · 验证完成标准 · 有问题启动新 session 修(**不要让原 team 改**)

- **跳过人 review 直接 push** → 5 个 agent 的产出没看就 push · 风险大

无人值守批量任务\
· 参考示例
=================

当一批小任务<u>不需要人盯着</u>(批量 refactor / 文档更新 / 翻译 /
注释生成)· 可以让 Claude 自己跑 · 你做别的事 · 结束后人 review。
下面是一个真实示例 · **不是必须按这个走**。

<div class="module-meta">

SCENARIO · **低风险批量任务 + 你不在场** STATUS · **团队尚未正式用过**

</div>

</div>

<div class="sub-section">

## 什么时候<u>可以考虑</u>无人值守

| ✅ 适合                         | ❌ 不适合 · 必须人在场       |
|---------------------------------|------------------------------|
| 批量更新 README / 注释          | 影响生产的 deploy            |
| 批量加测试覆盖率到现有代码      | 架构性变更                   |
| 批量 refactor 命名规范          | 需要判断的设计决策           |
| 批量翻译 / i18n 文本            | 需要 review 的安全敏感代码   |
| 清理 dead code · 没人调用的函数 | 跨模块依赖的 feature         |
| 批量 lint fix                   | 不可逆操作(数据库 migration) |

<div class="callout">

**核心判断**:如果某个任务失败了影响你睡得着觉吗?睡不着 → 不要 overnight
· 睡得着 → 可以。

</div>

</div>

<div class="sub-section">

## 一个真实启动示例

下面是一种可以参考的启动方式 · <u>不是唯一对的方式</u>。

<div class="step-card">

<div class="num">

示例 STEP 1

</div>

<div class="title">

睡前 30 分 · 挑 backlog + 写 task list

</div>

从飞书 backlog 挑 5-15 个适合无人值守的小任务 · 写到一个 markdown:

<div class="code-block">

docs/plans/overnight\_\[date\].md\
\
\## 今晚跑的 backlog\
\
- \[ \] Task 1: 把所有 console.log 换成 logger · 文件:src/\*\*/\*.ts\
- \[ \] Task 2: 给 utils/date.ts 加测试 · 覆盖率到 90%\
- \[ \] Task 3: 翻译 zh-CN 的 5 个新 UI 字符串\
\
完成标准:每个 task 跑测试通过 + 不动 src/auth/ + 不 push

</div>

</div>

<div class="step-card">

<div class="num">

示例 STEP 2

</div>

<div class="title">

设安全 guardrails

</div>

无人值守没人盯 · 必须有**明确的"不允许做什么"**:

- **不允许**:git push / npm publish / 改 .env / 改 .ssh / 数据库
  migration
- **允许**:写 src/ / 写 tests/ / 写 docs/ / 跑测试
- **遇到不确定**:跳过 · 写到 results 文件 · 不要硬上

具体怎么配置 guardrails(settings.json 之类) · 团队成员自己查 Anthropic
文档。

</div>

<div class="step-card">

<div class="num">

示例 STEP 3

</div>

<div class="title">

启动 + 设预算上限

</div>

让 Claude 按 task list 顺序执行 · 每个 task 写到 results 文件 · 任一
task 失败就跳过下一个。

**预算上限要设**(避免半夜烧光):

- 小 · 5 task → 建议 \$10-15
- 中 · 10-15 task → 建议 \$30-50
- 大 · 20+ task → 建议 \$80-150

具体 CLI 参数怎么设(`-p` mode / `--max-budget-usd`)·
团队成员自己查文档。

</div>

<div class="step-card">

<div class="num">

示例 STEP 4

</div>

<div class="title">

早上 review · 决定 merge / 丢弃

</div>

1.  读 results 文件 · 看每个 task 状态
2.  对每个完成的 task · 看 diff · 跑测试
3.  OK 的合并 · **人手 git push**(不要让 agent 自己 push)
4.  失败的单独处理 · 走 bug-fix playbook
5.  写 case file(<u>不可妥协 \#7</u>)· 记录:成功率 / 实际花费 /
    节省的时间

</div>

</div>

<div class="sub-section">

## 容易踩的坑(参考)

- **没设预算上限** → 早上来发现 \$300 没了
- **没禁止 git push** → agent 把错误代码 push 到 main
- **tasks 互相依赖** → 一个失败后面全错(应该改用多 agent 协同)
- **把生产敏感的事丢给无人值守** → 自找麻烦
- **不写 case file** → 没数据知道下次预算设多少
- **早上不亲眼看 diff 直接 merge** → 你 ship 的代码就是你的

- 周末家庭时间跑
- 跨时区协作时跑
- 等其他项目 review 时跑
